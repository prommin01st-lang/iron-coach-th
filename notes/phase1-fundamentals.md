# 📚 เฟส 1 — พื้นฐานที่ต้องรู้ (สรุปสำหรับทบทวน)

> ใช้ตัวอย่าง: บอทเวทเทรนนิ่ง 🏋️

## 1. Fine-tuning คืออะไร (เทียบกับวิธีอื่น)

โมเดลภาษา (LLM) ผ่าน 2 ช่วง:
- **Pre-training** — เรียนภาษา/ความรู้ทั่วโลกจากข้อความมหาศาล (ใช้ GPU หลายพันตัว — เราไม่ทำเอง)
- **Fine-tuning** — เอาโมเดลที่ pre-train แล้ว มา "ปรับเฉพาะทาง" ด้วยข้อมูลของเรา (เราทำขั้นนี้)

4 วิธีทำให้โมเดลเก่งงานเรา — เรียงจากเบาไปหนัก:

| วิธี | ทำอะไร | เหมาะเมื่อ | ตัวอย่างบอทเรา |
|---|---|---|---|
| **Prompting** | เขียนคำสั่งดี ๆ | ลองก่อนเสมอ | ใส่ system prompt "คุณเป็นเทรนเนอร์..." |
| **RAG** | ดึงข้อมูลจากเอกสารมาแปะให้โมเดลอ่านตอนตอบ | ต้องการ **ความรู้ใหม่/อัปเดตบ่อย** | ค้นเอกสารสรีรวิทยาแล้วให้โมเดลตอบจากนั้น |
| **Fine-tuning** | ปรับน้ำหนักโมเดลด้วยตัวอย่าง | ต้องการ **สไตล์/พฤติกรรม/รูปแบบตอบเฉพาะ** | สอนให้ตอบเป็นรูปแบบตาราง Sets/Reps เสมอ + น้ำเสียงเทรนเนอร์ |
| **Pre-training** | เทรนจากศูนย์ | แทบไม่มีใครทำเอง | ❌ ไม่ทำ |

**กฎง่าย ๆ:** อยากได้ "ความรู้" → RAG · อยากได้ "พฤติกรรม/สไตล์/รูปแบบ" → Fine-tune
(โปรเจกต์เราใช้ **fine-tune เป็นหลัก** และอาจเสริม RAG ทีหลังเพื่อกัน hallucination)

## 2. Transfer Learning — ทำไมไม่เทรนจากศูนย์
โมเดลรู้ภาษาไทย/อังกฤษ เหตุผล ตรรกะอยู่แล้ว เราแค่ "ต่อยอด" ความรู้เฉพาะทางเข้าไป
→ ใช้ข้อมูลน้อย (หลักร้อย–พันตัวอย่างก็เห็นผล) และ VRAM น้อยลงมาก

## 3. ศัพท์หลักที่เจอตอนเทรน

- **Token** — ชิ้นย่อยของข้อความ (คำ/ส่วนของคำ) โมเดลมองข้อความเป็นลำดับ token
- **Tokenizer** — ตัวแปลงข้อความ ↔ ตัวเลข (token id) แต่ละโมเดลมี tokenizer ของตัวเอง ต้องใช้ให้ตรงกัน
- **Epoch** — จำนวนรอบที่โมเดลอ่าน dataset ครบทั้งชุด (1 epoch = เห็นข้อมูลครบ 1 รอบ)
- **Batch size** — จำนวนตัวอย่างที่ป้อนพร้อมกันต่อ 1 ก้าว (VRAM น้อย → batch เล็ก)
- **Gradient accumulation** — สะสม gradient หลาย batch เล็กก่อนอัปเดต 1 ครั้ง = จำลอง batch ใหญ่โดยไม่กิน VRAM
- **Learning rate** — ก้าวการปรับน้ำหนักต่อครั้ง สูงไป = เรียนพัง, ต่ำไป = เรียนช้า
- **Loss** — ค่าความผิดพลาด ยิ่งต่ำ = ทายใกล้คำตอบจริง ดูให้มัน "ลดลงแล้วนิ่ง"
- **Overfitting** — ท่องจำ dataset จนตอบคำถามใหม่ไม่ได้ (train loss ต่ำ แต่ตอบจริงแย่)
- **Underfitting** — เรียนไม่พอ ยังไม่จับ pattern (loss สูงค้าง)

## 4. LoRA — หัวใจที่ทำให้เทรนบน 6GB ได้

ปัญหา: full fine-tuning = ปรับน้ำหนัก **ทุกตัว** (พันล้านตัว) → กิน VRAM มหาศาล

**LoRA (Low-Rank Adaptation)** = แช่แข็งโมเดลเดิมไว้ทั้งก้อน แล้ว **แปะเมทริกซ์เล็ก ๆ (adapter) เพิ่มข้าง ๆ** เทรนเฉพาะ adapter เท่านั้น
- เทรนแค่ ~0.1–1% ของพารามิเตอร์ทั้งหมด
- ไฟล์ผลลัพธ์เล็กมาก (หลัก MB) — สลับ adapter หลายตัวบน base เดียวได้
- พารามิเตอร์สำคัญ: **r** (rank ยิ่งสูง = จุมากขึ้นแต่กิน VRAM มากขึ้น, เริ่มที่ 8–16), **alpha** (สเกล, มักตั้ง = 2×r)

## 5. Quantization + QLoRA

**Quantization** = ย่อความละเอียดตัวเลขน้ำหนัก เช่นจาก 16-bit → **4-bit** ทำให้โมเดลเล็กลง ~4 เท่า เข้า VRAM ได้

**QLoRA = โหลด base model แบบ 4-bit (แช่แข็ง) + เทรน LoRA adapter ทับ**
→ นี่คือสูตรที่เราจะใช้ เพราะทำให้โมเดล 1.5B–3B เทรนได้สบายใน 6GB
(เราทดสอบ bitsandbytes 4-bit ผ่านแล้วในเฟส 0 ✅)

## 6. รูปแบบข้อมูลเทรน (Data Format)

โมเดล instruct/chat เรียนจากบทสนทนา รูปแบบมาตรฐาน = รายการ message:

```json
{"messages": [
  {"role": "system", "content": "คุณเป็นโค้ชเวทเทรนนิ่งที่ยึดหลักวิทยาศาสตร์การกีฬา"},
  {"role": "user", "content": "อยากเพิ่มความแข็งแรง Deadlift ควรจัด set/rep ยังไง"},
  {"role": "assistant", "content": "เป้าหมายความแข็งแรง: 3–5 เซ็ต × 3–5 ครั้ง ที่ ~80–90% 1RM ..."}
]}
```

เก็บเป็นไฟล์ `.jsonl` (1 ตัวอย่างต่อบรรทัด) โมเดลเรียนที่จะ "สร้างส่วน assistant" ให้เหมือนที่เราสอน

## 7. กัน Hallucination ด้วย dataset (ผูกกับโปรเจกต์เรา)
- คำตอบทุกอันต้องถูกต้องตามหลักวิทย์การกีฬา (โมเดลจะเลียนแบบสิ่งที่เห็น = garbage in, garbage out)
- ใส่ตัวอย่างที่ assistant ตอบว่า **"ข้อมูลนี้ควรปรึกษาผู้เชี่ยวชาญ/แพทย์"** เพื่อสอนให้รู้จัก "ปฏิเสธ" แทนการมั่ว
- ใส่คำเตือนความปลอดภัยในคำตอบที่เกี่ยวกับน้ำหนัก/อาการบาดเจ็บ
